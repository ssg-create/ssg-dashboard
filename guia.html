3
Como funcionam os filtros
Três filtros que trabalham juntos e se acumulam
⚠️
Os filtros são cumulativos. Se você selecionar Cliente + Mês + Fila ao mesmo tempo, o dashboard mostra apenas tickets que atendem às três condições juntas.
A barra colorida no topo da tela sempre mostra quais filtros estão ativos. Use o botão "limpar filtros" no canto superior direito para resetar tudo de uma vez.
💡
Dica rápida dentro do dashboard: ao lado de cada label de filtro na sidebar há um ícone ⓘ — clique nele para ver uma explicação rápida do que aquele filtro faz e como combiná-lo com os demais, sem precisar sair do dashboard.
📅
Filtro de Mês
Sidebar esquerda
Seleciona o período de análise. Por padrão mostra "Todos os meses" (acumulado do ano). Ao clicar em um mês, todos os KPIs, gráficos e contagens mudam para aquele período.
Todos os meses — soma acumulada do ano. O número ao lado do botão é o total geral.
Jan/26, Fev/26, Mar/26… — filtra apenas tickets criados naquele mês. O número ao lado mostra o total daquele mês.
Se um cliente estiver selecionado, os contadores de cada mês mostram apenas os tickets daquele cliente.
🗂️
Filtro de Fila
Sidebar esquerda
Restringe a análise a uma área técnica específica. Útil para líderes de squads que querem ver somente a sua área.
SSG PROTHEUS DATASUL GWMS DBA INFRAESTUTURA SSG-MELHORIAS
👤
Para gestores de área: ao filtrar pela sua fila (DBA, GWMS, etc), TODAS as métricas — inclusive Ações Prioritárias, Sem 1ª resposta, comparativos por cliente — passam a refletir só a sua fila. Cada gestor enxerga apenas o escopo que gerencia.
🏢
Filtro de Cliente
Rodapé da sidebar / aba Portal
Foca em um único cliente. Quando ativo, os KPIs da aba Operação também passam a mostrar apenas dados daquele cliente — combinado com Mês e Fila simultaneamente.
Todos os clientes — visão consolidada da operação.
Cliente selecionado — filtra ticket a ticket. O total exibido representa apenas aquele cliente no período/fila escolhidos.
Para limpar: clique em "limpar filtros" no canto superior direito da barra de status no topo da tela.

O que cada indicador significa
Referência rápida de todos os KPIs
💡
Mediana vs Média: O dashboard usa mediana no SLA de 1ª resposta — ela ignora outliers (um ticket esquecido por 5 dias não distorce o número). A média é usada no tempo de solução porque reflete o custo real de cada chamado.
IndicadorO que medeMeta / Referência
Total de chamados Tickets criados no período selecionado Base de comparação entre períodos
Em aberto Tickets ainda não concluídos (em atendimento, aguardando…) Ideal: abaixo de 15% do total
Taxa de resolução % de tickets concluídos com êxito no período Meta interna: ≥ 70%
Prioridade Muito Alta Tickets críticos abertos com prioridade máxima Requer atenção imediata se > 0
SLA 1ª Resposta Tempo até o primeiro contato com o cliente (mediana) Meta contratual: ≤ 30 min
Solução média Tempo médio de resolução dos tickets fechados Meta interna: ≤ 48h
Bloqueados Tickets parados — aguardando cliente, externo ou interno Alto número indica gargalo
Aguardando Cliente Tickets que dependem de ação do cliente para avançar Acompanhar semanalmente
Health Score (CS) Score 0–100: resolução + SLA + pendências + volume ≥70 Saudável · 50–69 Atenção · <50 Risco

Aba Operação Interna → seção Comparativo Mensal
2
Sem filtro de fila: ver o total consolidado por mês

Aplicar Fila = SSG → anotar os números → trocar para PROTHEUS → comparar
F
Comparar 2 períodos lado a lado (Mar/26 × Abr/26 ou Q1 × Q2)
1
Header da dashboard → botão Gerar Relatório PDF
2
No modal, escolher a opção Comparativo Mensal (4ª opção)

Como o sistema funciona por trás
Útil quando você precisa entender por que algo travou

Os dados que você vê no dashboard percorrem 5 etapas — cada uma rodando em um lugar diferente, todas automáticas. Saber a sequência ajuda a localizar onde olhar quando algo não bate.

OTRS/Znunybanco MySQL
→
cron-job.orgdispara a cada 10min
→
GitHub Actionsextrai + analisa
→
Repo de dadosJSONs salvos
→
Verceldashboard live
🏗️
Por que dois repositórios? O dashboard fica em um repo (ssg-dashboard), e os dados em outro (ssg-dashboard-data).

O sync grava ~720 vezes por dia — se gravasse no mesmo repo do site, a Vercel tentava re-publicar a cada gravação e estourava a quota gratuita em 3-7h. Separando, o repo de dados pode receber milhões de gravações sem afetar o site. Solução desenhada em 30/04/2026 — quota nunca mais é um problema.
⏱️
Quem dispara o sync
cron-job.org — serviço externo gratuito que faz uma chamada HTTP no GitHub a cada 10min, on-time (sem atraso). Backup interno do GitHub Actions cobre se o cron-job.org cair.
EXTERNO · GRATUITO · ON-TIME
⚙️
Quem extrai os dados
GitHub Actions — script Python que conecta no MySQL do OTRS via VPN/Grafana API, processa os tickets dos últimos 4 meses, gera os 8 JSONs e grava no repo de dados.
PYTHON · 4 MESES DE JANELA
📦
Onde ficam os dados
Repo público ssg-dashboard-data — 8 arquivos JSON: histórico completo, tickets ativos, utilização por atendente, insights ao vivo, silenciosos, triagem, reaberturas, aios.
8 JSONs · ATUALIZADOS A CADA 10MIN
🌐
Quem serve o site
Vercel — hospeda o HTML/CSS/JS e faz proxy dos JSONs (busca do repo de dados quando o navegador pede). Atualização do site é instantânea quando algum código muda.
VERCEL · CDN GLOBAL
🩺
Onde olhar quando algo trava (em ordem):
1. Banner vermelho "dados parados há ___ min" no topo do dashboard — confirma que o sync não rodou. Se aparecer, sintoma → causa nas etapas abaixo.
2. cron-job.org não está disparando → email pra Fabio Tavares (token vencido ou job desativado).
3. GitHub Actions falhou → erro no script Python (acesso ao MySQL, JSON malformado etc). Sistema envia email automático.
4. Vercel fora do ar → raríssimo, status em vercel-status.com.

Em 99% dos casos: causa #1 ou #2. Resolução típica em 5min.
🔒
Sobre privacidade: os JSONs ficam em repo público porque a Vercel precisa acessá-los sem autenticação. Não há credenciais, senhas, ou dados sensíveis dos clientes nos arquivos — só números, prazos e códigos de tickets do próprio OTRS interno. O acesso ao dashboard em si fica protegido por senha (ssg2026).
